# CV Pipeline

Personal CV tailoring pipeline. The master CV (`cv/main.tex`) is the source of
truth and is never trimmed. Tailored variants are generated by selecting a
subset of bullets from `bullet-bank/` against a job-ad spec in `job-ads/`.

## Ideology

The whole pipeline is shaped by five principles. Read these once — they
explain every other design choice in the repo.

1. **One page, high signal.** At early-career stage, density beats breadth.
   The tailored CV must compile to **exactly one page**, and every bullet on
   it must map to at least one keyword from the job ad. If a bullet doesn't
   earn its line, drop it.
2. **Master CV is canonical, never trimmed.** `cv/main.tex` is the
   kitchen-sink narrative — every real achievement lives there in full. We
   never delete from it to "make room"; we tailor *down* from it for specific
   roles.
3. **Bullets are reusable units, not prose.** Each bullet has a stable `id`,
   theme `tags`, and LaTeX-ready `text` in `bullet-bank/*.yml`. Tailoring is
   a *selection* problem (which IDs?), not a *rewriting* problem. Rewriting
   only happens when you genuinely improved a bullet — and then it lives in
   the bank for every future application.
4. **Specs, not edited PDFs.** A tailored CV is fully described by a small
   YAML spec in `job-ads/<company>.yml` (chosen bullet IDs + tailored profile
   + reordered skills). The `.tex` and `.pdf` are throwaway build artefacts.
   Iterate on the spec, never on the rendered output.
5. **Boundaries are non-negotiable.** No SONIQ source, no internal URLs, no
   confidential client names, no private metrics that aren't already public
   or safely generalised. The bullet bank is already generalised — keep it
   that way. See `BOUNDARIES.md`.

## Layout

```
cv-pipeline/
  cv/
    main.tex                    # master CV (kitchen-sink, never trimmed)
    tailored.tex.template       # template used by the renderer
  bullet-bank/
    soniq.yml                   # all SONIQ bullets, each with id + tags
    csiro.yml                   # all CSIRO bullets
    projects.yml                # projects + their bullets
  job-ads/
    _example.yml                # template for a job-ad spec
    <company>.yml               # one per role you apply to
  tools/
    render_tailored.py          # spec + bullet bank -> outputs/<company>.tex
  outputs/                      # generated .tex and compiled .pdf (gitignored)
  tools/
    render_tailored.py          # spec + bullet bank -> outputs/<company>.tex
    compile.sh                  # one-shot render + compile via Tectonic
  .cursor/skills/               # tailor-cv, add-bullet (Cursor Agent Skills)
  .vscode/                      # LaTeX Workshop preconfigured for Tectonic
  requirements.txt
  BOUNDARIES.md
```

## Setup

A virtualenv is already created at `.venv/` with PyYAML installed.

```bash
source .venv/bin/activate
# or, without activating:
.venv/bin/python tools/render_tailored.py job-ads/<company>.yml
```

To recreate from scratch:

```bash
python3 -m venv .venv
.venv/bin/pip install -r requirements.txt
```

### LaTeX compiler

[Tectonic](https://tectonic-typesetting.github.io/) is the recommended
compiler. It's a single Rust binary that fetches packages on demand on first
use, so it behaves like Overleaf without the upload step.

```bash
brew install tectonic
```

Alternatives: MacTeX (`brew install --cask mactex-no-gui`, ~6 GB) or BasicTeX
(`brew install --cask basictex` + `tlmgr install` extras). The `.tex` source
in this repo is engine-agnostic — it compiles under both pdflatex (Overleaf
default) and xelatex (Tectonic default). The pdfTeX-only ATS hint
(`\pdfgentounicode`) is wrapped in `\ifdefined` so it's used when present and
silently skipped otherwise.

### In-Cursor preview (LaTeX Workshop)

`.vscode/settings.json` is preconfigured to use Tectonic via the
[LaTeX Workshop](https://marketplace.visualstudio.com/items?itemName=James-Yu.latex-workshop)
extension. When you open any `.tex` file in this repo, Cursor will prompt to
install LaTeX Workshop (it's listed in `.vscode/extensions.json`). After
installing:

- Save a `.tex` file → it auto-compiles.
- `Ctrl+Alt+V` (or the eye icon) → open the PDF in a side tab.
- The PDF live-reloads on every save.

### One-shot compile

`tools/compile.sh` renders + compiles in a single command:

```bash
tools/compile.sh job-ads/<company>.yml   # render YAML spec then compile
tools/compile.sh outputs/<company>.tex   # compile an existing .tex
tools/compile.sh master                  # compile cv/main.tex (full master)
```

It warns if a tailored CV exceeds one page.

## Primary workflow: use the agent

The fastest way to use this repo is through the two Cursor Agent Skills.
Open Cursor in this folder and just talk to it — the skills auto-trigger
from natural phrasing.

### Skill 1: `tailor-cv`

Paste a job ad, the agent picks bullets against it, you approve, it writes
the spec, renders, and compiles. Targets one page, signal-first.

**Template prompt — copy, paste, fill in:**

```
Tailor my CV for this role.

Company: <Company name>
Role title: <e.g. Software Engineer, Backend Engineer, Graduate SWE>
Source URL: <link to the ad, or "n/a">

--- AD START ---
<paste the full job description verbatim here>
--- AD END ---

Notes (optional):
- Anything I want emphasised: <e.g. "lean into AWS + IaC">
- Anything to downplay: <e.g. "skip the mobile/Android stuff">
- Page target: 1 page (default)
```

The agent will then:
1. Read the bullet bank + `BOUNDARIES.md`.
2. Extract keywords / must-haves / culture signals from the ad.
3. Show a shortlist table (`Bullet ID | Why | Keyword(s) hit`) and a draft
   `profile` paragraph and `skills` block.
4. **Wait for your approval** before writing files.
5. Write `job-ads/<slug>.yml`, run the renderer, compile to PDF.
6. Enforce one page (auto-prune if it overflows; recommend additions if
   under-full).
7. Offer iteration: "swap X for Y", "shorten profile", etc.

### Skill 2: `add-bullet`

You did something at work or on a project — log it before you forget.

**Template prompt:**

```
Add this to my bullet bank.

Bank: <soniq | csiro | projects:<project-id> | new-project>
Raw description:
  <free-form: what I built, why, the outcome, any numbers>
Mirror to cv/main.tex? <yes | no | ask me later>
```

The agent will:
1. Polish the wording in house style (strong verb, outcome-oriented, ≤ 25 words).
2. Assign a stable `id` (kebab-case, prefixed with the bank) and 3–6 `tags`.
3. Show the polished bullet for **your approval**.
4. Append to the right `bullet-bank/*.yml` and (optionally) mirror to
   `cv/main.tex` between the role's last bullet and its `\textbf{Stack:}` line.

If your raw description has no concrete metric, the agent will ask before
inventing one — it never fabricates numbers.

## Manual workflow (no agent)

If you'd rather drive it by hand:

1. **Save the ad.** Copy `job-ads/_example.yml` to `job-ads/<company>.yml`
   and paste the raw description into `ad_raw`. Fill in `keywords`.
2. **Pick bullets.** For each role, list bullet `id`s from the bank.
   Budget for one page: 4–5 SONIQ + `soniq-stack`, 3–4 CSIRO +
   `csiro-stack`, 2 projects with 1–2 bullets each. Total `\resumeItem`
   lines ≤ 16.
3. **Render + compile.**
   ```bash
   tools/compile.sh job-ads/<company>.yml
   open outputs/<company>.pdf
   ```
   Or in two steps:
   ```bash
   .venv/bin/python tools/render_tailored.py job-ads/<company>.yml
   tectonic outputs/<company>.tex --outdir outputs
   ```
5. **Iterate.** If > 1 page, drop overview bullets first, then the lowest-
   keyword-overlap bullets, then weak projects. Re-render. Never edit
   `outputs/*.tex` by hand.

## Adding a new bullet (manual)

Append to the relevant `bullet-bank/*.yml` with a unique `id`, `tags`, and
LaTeX-ready `text`. The bullet is then available to every future spec.

## What stays in `main.tex`

The master is the canonical narrative. Update it when you have a real new
achievement, then mirror it into the bullet bank so tailored versions can
pick it up. The `add-bullet` skill does both in one step.

## Boundaries

See `BOUNDARIES.md`. Short version: no SONIQ source, no internal URLs, no
confidential client names, no private business metrics unless already
approved/public or safely generalised. Bullets in the bank are already
generalised — keep it that way.
